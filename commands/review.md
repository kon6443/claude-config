변경된 코드에 대해 플로우 기반 QA 리뷰를 수행합니다.
모든 판단은 근거 기반(코드 직접 확인)으로만 진행합니다. 추측/추론 금지.

## 1단계: 변경 파일 파악

`git diff --name-only`로 변경 파일 목록을 확인합니다.
변경 파일에서 추가/수정된 함수, API, 타입을 식별합니다.

## 2단계: grep 전수 확인

변경된 함수/API/타입 이름으로 프로젝트 전체 grep을 실행하여 **모든 호출 위치**를 파악합니다.
- 동일 함수가 여러 곳에서 호출되면 중복/불필요 여부 교차 비교
- 파일 부분 읽기(offset/limit)만으로 "전체 정상" 판단 금지
- 변경된 이벤트/상태명으로 **수신자(리스너/핸들러/구독자/캐시 무효화)**도 전수 파악.
  변경 파일 밖의 기존 핸들러가 이번 변경 맥락에서 충분한지 교차 검증.

## 3단계: 플로우 리스트업

변경 내용을 유저 액션 기준 플로우로 분류합니다.
각 플로우: `유저 액션 → 핸들러 → API/서비스 → DB → 응답/이벤트`

## 4단계: 플로우별 코드 추적 검증

각 플로우마다:
1. **중복 호출**: 동일 API/함수가 불필요하게 여러 번 호출되지 않는지
2. **타입 일치**: 데이터가 전달되는 모든 경로에서 타입/필드가 일치하는지
3. **필드 누락**: 새로 추가한 필드가 전체 체인에 포함되는지
4. **기능 정합성**: 의도한 데이터만 변경되고, 불필요한 부수효과 없는지
5. **부수 효과 전파**: 변경된 흐름이 알림/이벤트/로그/메트릭/소켓 브로드캐스트 등 외부 시스템으로 나가는 경우, 변경된 조건(예: 필터 제거·권한 완화) 하에서 그 부수 효과가 **의도한 범위**로 확장됐는지 정책 판단. 확장이 부적절하면 플래그.
6. **대칭 분기 검증**: 상태 전이가 쌍으로 구성되는 경우(생성↔삭제, 활성↔비활성, 로그인↔로그아웃, 추가↔제거), **반대 분기가 동일 품질로 처리됐는지** 확인. 한쪽만 구현된 핸들러·캐시 무효화·토스트는 플래그.
7. **관련 핸들러 완결성**: 변경된 도메인을 다루는 **기존 리스너/핸들러/캐시 경로**(변경 파일 밖이어도)가 이번 변경의 맥락에서 여전히 충분한지. 2단계에서 파악한 수신자 목록을 기반으로, 각 수신자가 변경 후에도 정상 반응하는지 코드 직접 확인.

## 5단계: 중복 코드 / 리팩토링

- 동일 패턴이 3곳 이상 반복되면 공통 함수 추출
- 오버엔지니어링 금지 — 추출이 코드를 더 복잡하게 만들면 그대로 둠

## 6단계: 빌드 검증

프로젝트 빌드 커맨드를 실행하여 타입 에러 없는지 확인합니다.
