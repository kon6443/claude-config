변경된 코드에 대해 플로우 기반 QA 리뷰를 수행합니다.
모든 판단은 **측정·실행 기반**으로만 진행합니다.
- 근거 = 코드 직접 확인 · grep 전수 결과 · 실제 실행/측정 결과. 이외는 근거 아님.
- 추측/추론/대략 금지. "~정도", "~쯤", "~일 것", "대략 Xpx" 등의 표현은 **자동 불합격**.
- 수치는 반드시 실제 값으로 판단:
  - CSS 수치: padding/font-size/line-height를 명시적으로 합산하거나 computed value 확인
  - 응답/타이밍: 실제 측정 결과
  - 영향 범위: grep 결과 파일 목록 전체 Read 후 판정

## 1단계: 변경 파일 파악

`git diff --name-only`로 변경 파일 목록을 확인합니다.
변경 파일에서 추가/수정된 함수, API, 타입을 식별합니다.

## 2단계: grep 전수 확인

변경된 함수/API/타입 이름으로 프로젝트 전체 grep을 실행하여 **모든 호출 위치**를 파악합니다.
- 동일 함수가 여러 곳에서 호출되면 중복/불필요 여부 교차 비교
- 파일 부분 읽기(offset/limit)만으로 "전체 정상" 판단 금지
- 변경된 이벤트/상태명으로 **수신자(리스너/핸들러/구독자/캐시 무효화)**도 전수 파악.
  변경 파일 밖의 기존 핸들러가 이번 변경 맥락에서 충분한지 교차 검증.

## 3단계: 플로우 리스트업

변경 내용을 유저 액션 기준 플로우로 분류합니다.
각 플로우: `유저 액션 → 핸들러 → API/서비스 → DB → 응답/이벤트`

## 4단계: 플로우별 코드 추적 검증

각 플로우마다:
1. **중복 호출**: 동일 API/함수가 불필요하게 여러 번 호출되지 않는지
2. **타입 일치**: 데이터가 전달되는 모든 경로에서 타입/필드가 일치하는지
3. **필드 누락**: 새로 추가한 필드가 전체 체인에 포함되는지
4. **기능 정합성**: 의도한 데이터만 변경되고, 불필요한 부수효과 없는지
5. **부수 효과 전파**: 변경된 흐름이 알림/이벤트/로그/메트릭/소켓 브로드캐스트 등 외부 시스템으로 나가는 경우, 변경된 조건(예: 필터 제거·권한 완화) 하에서 그 부수 효과가 **의도한 범위**로 확장됐는지 정책 판단. 확장이 부적절하면 플래그.
6. **대칭 분기 검증**: 상태 전이가 쌍으로 구성되는 경우(생성↔삭제, 활성↔비활성, 로그인↔로그아웃, 추가↔제거), **반대 분기가 동일 품질로 처리됐는지** 확인. 한쪽만 구현된 핸들러·캐시 무효화·토스트는 플래그.
7. **관련 핸들러 완결성**: 변경된 도메인을 다루는 **기존 리스너/핸들러/캐시 경로**(변경 파일 밖이어도)가 이번 변경의 맥락에서 여전히 충분한지. 2단계에서 파악한 수신자 목록을 기반으로, 각 수신자가 변경 후에도 정상 반응하는지 코드 직접 확인.
8. **Ripple Check (변경 속성 ↔ 미변경 속성 간 상호작용 재검증)**: 이번 변경 E로 인해 **E의 변경되지 않은 속성**이 **E의 이웃 N**과 맺는 관계가 달라지는 경우를 포함해 재확인.
   - **이웃 N 식별 (도메인별)**:
     - FE: 같은 viewport/컨테이너에서 공간적으로 충돌 가능한 fixed/absolute/sticky 요소, 같은 z-index 층, 같은 레이아웃 컨테이너 형제·조상, 같은 state/store 구독자
     - BE: 변경된 심볼명으로 grep해서 나오는 모든 caller/callee, 같은 이벤트·DB 엔티티·캐시 키·인덱스를 공유하는 경로, 동일 미들웨어/가드가 적용되는 라우트 집합
   - **재검증 요점**: E의 변경된 속성 + 미변경 속성이 함께 작용해, N의 기존 불변식(layout invariant · API contract · 성능 가정 · 권한 경계)을 여전히 성립시키는지
   - **맹점 패턴**:
     - FE: "bottom 값은 유지됐지만 right가 바뀌며 다른 fixed 요소와 공간 겹침 발생"
     - BE: "응답 스키마는 유지됐지만 enum 값 의미·nullability가 바뀌어 consumer 해석 달라짐", "where 조건은 바뀌었는데 해당 컬럼 인덱스가 없어 다른 쿼리 성능까지 저하"
   - **판정**: "변경 안 건드림 → OK" 자동 종결 금지. 반드시 이웃과의 상호작용까지 확인 후 판정.

## 5단계: 중복 코드 / 리팩토링

- 동일 패턴이 3곳 이상 반복되면 공통 함수 추출
- 오버엔지니어링 금지 — 추출이 코드를 더 복잡하게 만들면 그대로 둠

## 6단계: 실행 검증 (추정 금지, 실제 실행만 인정)

- **공통**: 빌드/타입체크 실행 후 에러 0건 확인 (명령어와 결과를 출력에 명시)
- **UI 변경 포함 시 (프론트엔드)**:
  - 검증 viewport는 **변경 영향 범위**에 맞춰 선택:
    - 반응형 / 레이아웃 컨테이너 / 전역 레이아웃 관련 → 모바일(≤480px) + 태블릿(~768px) + 데스크탑(≥1280px) 3종 모두
    - 특정 단일 브레이크포인트에서만 렌더되는 컴포넌트 → 해당 브레이크포인트만
    - 불확실하면 → 3종 모두
  - dev server가 **이미 기동 중이면** 사용자에게 실제 렌더 확인을 **명시적으로 요청** (AI가 새로 기동하지 말 것)
  - dev server가 **꺼져 있으면** 기동 여부를 **사용자에게 먼저 확인**
  - 실물 확인 없이 "시각 이슈 없음" 단정 금지. 확인 불가 시 "시각 검증 미수행, 사용자 확인 필요"로 명시
- **API/DB/이벤트 변경 포함 시 (백엔드)**:
  - 관련 단위/통합 테스트 실행. 없으면 smoke 테스트 1개 추가 제안
  - DB 쿼리 변경 시 실행 계획(EXPLAIN) 또는 인덱스 사용 여부 확인
